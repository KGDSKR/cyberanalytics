# CyberAnalytics — Telegram Mini App

Отвечай по-русски. Пользователь (Вилли) — контент-креатор, не программист: делегирует всю
техническую часть полностью («ты мои руки», роль Claude — техлид с полной автономией).
Объясняй, что куда класть и что нажимать, давай решения сразу и по шагам.

## Что это

Мини-приложение Telegram: ИИ-аналитика киберспортивных матчей CS2 (позже Dota 2).
Пользователь видит список ближайших матчей, жмёт на матч → Claude генерирует анализ
(прогноз победы, форма, составы, личные встречи, риски). Справочно — ссылка на Betboom.
Анализ информационный, не букмекерский сервис: дисклеймер «не финансовая рекомендация, 18+».

## Архитектура

npm workspaces, две части:

- **`backend/`** — Node.js 22 + Fastify + TypeScript (запуск через tsx, без сборки).
  Прячет ключи API, отдаёт REST: `GET /api/matches`, `POST /api/analyze {matchId}`,
  `GET /api/health`. В проде также раздаёт статику `frontend/dist`.
  - `pandascore.ts` — клиент PandaScore. Обе игры: CS2 (слаг `csgo` в их API!) и Dota 2.
    Матчи: live (`/matches/running`) + предстоящие (`/matches/upcoming`), по 50 на игру.
  - `ai.ts` — два провайдера, выбор через env `AI_PROVIDER`: **gemini** (основной,
    REST-вызов `generateContent`, модель `GEMINI_MODEL`) и claude (запасной,
    `@anthropic-ai/sdk`, adaptive thinking). Без ключей → мок-анализ.
    **Важно:** новые Gemini-модели на бесплатном тарифе часто отвечают 503 «high
    demand» — в `generateWithGemini` встроен ретрай + фолбэк-цепочка моделей
    (`GEMINI_FALLBACKS`). Не убирать.
  - `telegram.ts` — HMAC-проверка `initData` (включается `REQUIRE_TG_AUTH=true`).
  - `dota-live.ts` — live-драфт текущей карты Dota 2 (герои, киллы, золото). Источники:
    Steam Web API (`STEAM_API_KEY`, полное покрытие лиг) + OpenDota `/live` как запасной
    (без ключа, видит только топ-игры). Матчинг с PandaScore — по названиям команд
    (нормализация в `norm()`). Драфт показывается плашкой в UI (`GET /api/draft`) и
    подаётся в промпт Gemini для учёта в вероятностях — БЕЗ отдельного раздела в анализе
    (требование Вилли). Live-анализ протухает через 10 мин или при смене счёта.
  - Готовые анализы сохраняются в `data/analyses/{matchId}.json` — задел под
    метрику «точность прогнозов постфактум».
- **`frontend/`** — React + Vite + TypeScript. Тёмная киберспортивная тема
  (фон #0b0e14, акцент #c4f82a, шрифты Russo One + Manrope — оба с кириллицей).
  Telegram WebApp API подключён через `<script>` в index.html (без npm-SDK).
  Markdown анализа рендерится через `marked`.

## Ключевые правила

- **Ключи только в `.env`** (в корне, gitignored). Никогда не коммитить и не зашивать в код.
- **Демо-режим**: без ключей всё работает на моках (`backend/src/mock-data.ts`) —
  фронт показывает баннер «Демо-данные». Это осознанное поведение, не баг.
- Guardrail в промпте ИИ: никаких советов по суммам и призывов ставить — не убирать
  (вероятностные оценки рынков — можно, это суть продукта).
- **Стиль анализа (обновлено 2026-07-03)**: не телеграфная сводка, а настоящее
  рассуждение — модель обязана объяснить ПОЧЕМУ, а не просто переложить винрейт
  в проценты. В `systemPrompt()` зашиты: обязательный шаг «адвокат дьявола» перед
  финальной цифрой, потолок калибровки (фаворит редко >75-80% на реальных данных,
  выше — только при почти полном доминировании), дисциплина при маленькой выборке
  (<5-6 матчей → сужать разрыв), чёткая граница «из данных» vs «общие знания как
  фон» (никогда не выдумывать составы/ранги/тиры соперников). Не упрощать обратно
  в сухую таблицу без рассуждения — это осознанный откат от более раннего
  «телеграфного» стиля, менять только по прямому запросу Вилли.
  Обязательный блок «Рынки»: тотал карт 2.5, фора ±1.5, точный счёт, для Dota 2 — тотал
  по времени карты. Агрегаты (винрейты, серии, средние длительности) считаются в коде
  (`teamAggregates` в `ai.ts`), модели арифметику не доверяем. Рынки без данных (тотал
  киллов, тотал раундов CS2) модель помечает «нет данных» — для них нужен платный тариф
  PandaScore или другой источник.
- **Сигналы точности (добавлено 2026-07-12), все бесплатные:**
  - `tournament.tier`/`prizepool` — PandaScore отдаёт это БЕСПЛАТНО (не за `detailed_stats`,
    в отличие от раундов/карт/киллов). Прокинуто в `Match` и `PastMatchSummary` (`tier`),
    используется и для текущего матча, и для каждого матча в истории команды —
    `teamAggregates()` считает разрез побед/поражений топ-турниры (S/A/B) vs остальные.
    Модель обязана опираться на эти буквы, а не гадать «тир-3 оппонент» от балды.
  - `playedEarlierToday()` в `ai.ts` — усталость (второй матч в один день) вычисляется
    в коде из уже загруженной истории (`recentByTeam`), без доп. запросов к API.
    Юнит-тест на 5 граничных случаев (сегодня/вчера/будущее/пусто/смешанный список)
    прогонялся вручную перед деплоем — логика подтверждена.
  - Состав игроков (список 5 активных ников с `/teams/{id}`) — сознательно НЕ подключали:
    без даты трансфера и без per-match lineup (тоже платный тариф) это просто текст без
    проверяемой пользы, разбор см. в истории переписки 2026-07-12.
- **Калибровка на реальных данных (2026-07-12)**: скачаны все прогнозы из ветки `data`
  (на тот момент 210), сверены с реальными итогами через PandaScore. Общая точность
  56.2% (113/201 решённых). Ключевая находка: диапазон уверенности 70-80% сбывался лишь
  в 54.2% случаев (n=59) — почти монетка при уверенном виде; 80%+ отрабатывал честно
  (80-86%, но выборка меньше — n=10/n=7). Потолок обычной уверенности в промпте снижен
  до 65-68%, зона 80%+ оставлена только для действительно многосигнального доминирования.
  Стоит повторить этот замер через несколько недель на новых прогнозах (написанных уже
  этим промптом) — сравнить, помогла ли правка, и не тюнить калибровку заново вслепую.
- **Gemini — качество и квоты (важно, не наступать повторно)**: Pro-модели
  (`gemini-3.1-pro` и т.п.) на бесплатном ключе имеют лимит **0** запросов — полностью
  недоступны без оплаты, даже не пытаться. Лучшая бесплатная модель — Flash-класс;
  но `gemini-3.5-flash` (топ по качеству) на free tier — всего ~20 запросов/сутки,
  дальше 429 с `PerDay` в теле ошибки. Поэтому в `GEMINI_FALLBACKS` (`ai.ts`) — цепочка
  из моделей с намного более щедрой квотой (`gemini-3-flash-preview`, `gemini-2.5-flash`,
  `gemini-2.5-flash-lite`): именно они тянут основной объём автопрогнозов, флагман
  используется по возможности первым. `thinkingConfig.thinkingBudget` зафиксирован на
  максимуме Flash-уровня (24576, не `-1` динамический — модель обязана выработать весь
  бюджет). **Критично**: `maxOutputTokens` обязан быть заметно больше thinking-бюджета
  (сейчас 32768) — иначе задокументированный баг Gemini 2.5/3: размышление съедает весь
  лимит токенов и в ответе приходит пустая строка. Не понижать без явной причины.
- Анализ live-матча пересчитывается при смене счёта (сравнение в `routes.ts`).
- **Трекинг точности**: ИИ добавляет в конец ответа скрытый `<!--PRED {"team1":X,"team2":Y}-->`,
  он вырезается (`extractPrediction` в `ai.ts`) и первый прогноз по матчу сохраняется
  навсегда в ветку **`data`** GitHub-репо (`github-store.ts`; диск Render эфемерный!).
  `GET /api/accuracy` сверяет прогнозы с итогами PandaScore; вкладка «Точность» в UI.
- **Безопасность**: подпись initData отвергается при `auth_date` старше 24 ч;
  rate-limit 60 req/мин на IP (на `/api/analyze` — 5/мин); markdown ИИ проходит
  DOMPurify перед рендером. Не ослаблять.
- PandaScore free tier: 1000 запросов/час — кэш матчей 60 сек в `cache.ts` обязателен.

## Отловленные грабли — не наступать повторно

1. **Google блокирует бесплатный Gemini API с европейских дата-центровых IP** (403 с
   HTML-страницей). Из-за этого сервис переехал из Frankfurt в Oregon (2026-07-03,
   старый `srv-d93e5si8qa3s73ah2l20` удалён). С домашних IP работает, с EU-хостингов — нет.
   Регион сервиса не менять на европейский.
2. PandaScore: отменённые матчи имеют `end_at=null` и при `sort=-end_at` всплывают
   наверх — обязательны `filter[status]=finished` + `range[end_at]` (уже в коде).
3. OpenDota `proMatches` отстаёт от реальности на несколько часов — драфты совсем
   свежих завершённых матчей временно «не найдены», это норма. Условие остановки
   листания ленты — по max(start_time) страницы, не по min (единичные старые записи).
4. Классический GitHub-токен без scope `workflow` не может пушить `.github/workflows/`.
5. Кавычки внутри `-m` при вызове git из PowerShell ломают аргументы — в сообщениях
   коммитов не использовать `"`.
